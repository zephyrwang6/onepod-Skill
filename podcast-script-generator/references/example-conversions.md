# 转换示例

## 示例一：AI产品方法论

### 输入（结构化笔记）

```
1、为六个月后的模型做产品。Boris 说他们不为"今天的模型"做产品，而是盯住六个月后的能力边界。短期看是等待，但长期看能避免把时间浪费在很快被模型进化抹平的细小优化上。

2、Claude Code 是从终端聊天开始的。最初只是为了理解 API，做了一个终端里的聊天应用。工具使用出来后，他把批处理工具接上去，意外发现模型可以读文件、甚至用脚本查询本机音乐播放，这成为他第一次"这就是 AGI 的味道"的时刻。
```

### 输出（口播脚本）

```
今天要给你推荐一期播客，Boris Cherny 在 Y Combinator Lightcone 的最新访谈。这期是我最近听到的对AI产品开发最有深度的一期。

介绍一下 Boris，他是 Anthropic 的工程师，也是 Claude Code 的创造者。他对"模型能力进化"有非常深刻的洞察。这期节目聊了很多关于AI产品怎么做的话题，我挑几个最反直觉的点跟你分享。

第一，Boris 说他们做 Claude Code 完全不是在为"今天的模型"做产品，而是在盯着六个月后的能力边界。

为什么？因为他们发现，现在花时间优化那些当前模型做不好的地方，六个月后模型自己就变强了，这些优化全都被抹平了。他们宁愿等待模型进化，而不是堆功能。我觉得这个策略特别值得思考。

第二，关于 Claude Code 的起源故事也特别有意思。

最开始做这个工具，其实只是为了理解 API，想最快搭一个可以用 API 的东西。终端最省事，不需要 UI，所以就从终端聊天开始。但当他把批处理工具接上去之后，意外发现模型可以读文件、甚至用脚本查询本机音乐播放。那一刻他说，这是他第一次感受到"这就是 AGI 的味道"。

很夸张的是，他原本以为终端只是起点，没想到它就这样被保留下来了。在模型快速进化时期，任何复杂 UI 都可能在半年内过时，终端这种简单的约束反而让体验变得简单、可用、好玩。
```

## 示例二：代理经济趋势

### 输入（结构化笔记）

```
1、从"自动补全"进入"代理决策"。一年之前还是 Cursor vs Windsurf 这样的高级自动补全，现在 Claude Code 让人开始信任代理独立决策，多个代理并行执行任务。

2、Moltbook 是"代理社会"的雏形。Agent-only 社区让人第一次看到"没有人类参与的互动"，提示未来内容生产和互动的主体可能是模型。

3、代理会选择工具，带来新的经济系统。工具不再只是人类口碑驱动，而是代理根据文档和 API 体验自己"选型"，形成平行于人类市场的代理经济。
```

### 输出（口播脚本）

```
今天要给你推荐一期播客，Y Combinator 的 Lightcone 最新一期节目，聊的是"AI Agent Economy"代理经济。

这期是我最近听到的对开发者工具未来最有前瞻性的一期。他们聚焦在 OpenClaw、Moltbook 等现象，讲清楚了一个正在发生的趋势：开发者工具的购买者正在从"人"变成"代理"。

我挑几个核心观点跟你聊。

第一，我们从"自动补全"进入了"代理决策"时代。

一年之前大家还在讨论 Cursor vs Windsurf 这样的高级自动补全，现在 Claude Code 已经让人开始信任代理独立决策，多个代理并行执行任务。这意味着什么？产品服务的对象正在从人类开发者迁移到代理。

第二，Moltbook 是"代理社会"的雏形。

这是一个 Agent-only 社区，让人第一次看到"没有人类参与的互动"。你可以想象，未来内容生产和互动的主体可能真的会是模型，而不是人。Moltbook 就像一个预演版的"代理社会"。

第三，代理会选择工具，带来新的经济系统。

工具不再只是人类口碑驱动，而是代理根据文档和 API 体验自己"选型"。这形成了一个平行于人类市场的代理经济。特别反直觉的是，Supabase 走红的背后原因，竟然是它的文档更容易被模型理解和执行。

节目中还举了一个反面例子：Claude Code 选择了 Whisper v1 做转录，速度慢又贵，但更快的方案因为文档复杂而被代理忽略了。这说明在代理经济时代，文档从"说明书"变成了"增长渠道"。
```

## 转换技巧总结

### 1. 开场多样化

| 内容特点 | 开场示例 |
|---------|---------|
| 非共识观点多 | "这期是我最近听到的对[领域]非共识观点最多的一期..." |
| 深度洞察 | "这期是我最近听到的对[话题]最有深度的一期..." |
| 前沿趋势 | "这期讲了[话题]最前沿的几个趋势..." |
| 实用方法论 | "最近听到一期特别干货的节目..." |
| 人物故事 | "今天要给你推荐一期[嘉宾]的访谈..." |

### 2. 自然过渡（替代"不是...而是"）

❌ 禁用：
- "不是设计领域主动想变，而是工程师的变化倒逼着设计改变"
- "不是...而是..."
- "是...不是..."
- "从...变成...，从...变成..."

✅ 改用：
- "设计不是主动想变，是工程端的变化逼着你必须变"
- "以前大家把这套流程当真理，但现在基本已经死了"
- "工程师已经转向命令行和agent模式，而设计师开始用IDE了"
- "以前要花大部分时间画稿，现在只剩30-40%"

### 3. 解释概念（少用"说白了"）

❌ 避免：
- "说白了，这就是..."
- "说白了就是..."

✅ 改用：
- 直接解释："为什么？因为..."
- 用场景解释："举个例子..."
- 用结果说明："结果就是..."
- 或干脆省略，直接说结论

### 4. 观点展开公式

结构化笔记 → 口播脚本：

```
[自然引入] → [核心观点] → [解释/展开] → [个人评价]

示例：
"第一，Boris 说他们做产品完全不是在为今天的模型做，而是在盯着六个月后的能力边界。

为什么？因为现在优化的东西，六个月后模型自己就变强了，这些优化全都被抹平了。他们宁愿等待模型进化，而不是堆功能。我觉得这个策略特别值得思考。"
```

### 5. 强调技巧

- "很夸张的是" —— 用于强调数据/事实
- "特别反直觉的是" —— 用于反常识观点
- "特别有意思的是" —— 用于有趣发现
- "让我特别意外的是" —— 用于情感共鸣

### 6. 段落节奏
- 每段2-4句话
- 段落间留空行
- 一个段落一个观点
- 重要数据/金句单独成段

### 7. 禁用清单

- ❌ "发言人" 标记
- ❌ "不是...而是..."
- ❌ "是...不是..."
- ❌ "被...倒逼..."
- ❌ 对仗工整的书面化对比
- ❌ 过多的"说白了"
